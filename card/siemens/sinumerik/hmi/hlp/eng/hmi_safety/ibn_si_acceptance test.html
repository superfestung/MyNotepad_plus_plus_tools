Test types used

"Overview" setting area

The "Overview" setting area is automatically activated after the acceptance test function is called. In this setting area, you can either manually enter the required machine data, or load the content of a template ("Load template" softkey) to the setting area. The following machine data can be entered:

  • Machine designation

  • Machine type

  • Serial number

  • Machine manufacturer

  • PLC manufacturer

  • End user

  • Name of the test engineer

You can also select the following identifies as option:

  • Series commissioning

  • Test because of SI modification

"F-PLC project data" setting area

Checklist to verify the correct implementation of safety-relevant project data. Using the safety printout from the TIA Portal, various settings must be checked, e.g. whether the hardware is configured correctly. The test type is only intended for documentation.

"Switch-off signal path" setting area

You can test the switch-off signal paths in the "Switch-off signal path" setting area. Follow the instructions in the detailed view.

You can make the following settings here:

  • Test name

  • Criteria for initiating and evaluating the test

  • Acquired alarms that should appear in the report

  • Evaluating the test results

Note

Display of relevant alarm messages at SINUMERIK Operate

The forced checking procedure of the switch-off signal paths can be carried out simultaneously for several axes/drives. Depending on the communication load, it cannot always be guaranteed that all alarms are displayed at the SINUMERIK Operate.

Remedy:

  • Restrict the number of axes/drives that are simultaneously tested. Recommendation: 6 axes/drives.

  • Alternatively, adapt the time for checking the pulse inhibit (p9557 SI Motion STO test time).

"Function relationships" setting area

User-specific test cases for safety-related functions can be saved in the "Functional relationships" setting area. The required steps are then to be carried out by the user at the plant after the test case has been started, the behavior observed, and the observed test result documented.

A function table serves as basis when testing the "Functional relationships". Example:

Machine state

Safety state of the spindle, MSP1

Safety state of the feed axes, MX1, MY1, MZ1

Door closed and locked

SLS4

SLS4

Door open

STO

SOS

Door open, "Setup" mode and agreement button pressed

STO

SLS2

In the test, for each machine state, a check is made as to whether the spindle and feed axes transition into the safety state as defined in the function table.

You can make the following settings here:

  • Test name

  • Criteria for initiating and evaluating the test

  • Acquired alarms that should appear in the report

  • Evaluating the test results

Note

When creating a test instance, a user-defined trace can be activated, which documents the signal characteristic of the signals that are relevant for the particular test case.

The trace is parameterized using the standard SINUMERIK Operate function and saved as trace template (XML file).

"Position adjustment" setting area

For this test type, you can verify the agreement between the SI actual value sensing and the higher-level control system (e.g. SINUMERIK) or the standard encoder evaluation of the drive.

For a machine with SI monitoring functions, before safety commissioning, the axis/spindle is commissioned without SI function. The encoder evaluations of the machine are measured, and are therefore used as reference for the SI encoder evaluation.

For the SI acceptance test, a check must be made as to whether the SI positions/velocities from Safety Integrated agree with those of the "closed-loop control".

"Safety Activation Test" setting area

During the "Safety Activation Test", the logic of the safety program is tested and documented. The interrelationship between safety-relevant sensors via the safety program up to the active monitoring function of the axis/drive and actuator control is checked.

All of the signals relevant for a safety function are written in to an input or response table. The start-end signal states required for the test are specified and checked in the test workflow.

You can assign variables or signals to the Safety Activation Test. The following options are available:

  • Enter the signal in the dialog

  • Select a variable from the list of internal variables (e.g. system variables)

  • Read in PLC signals from the TIA Portal and select signals

"Wiring test" setting area

The input and output signals are tested when performing a wiring test. During the test sequence, the status of signals and feedback signals is changed.

The acceptance test identifies the status change and acknowledges this. The individual signals are tested in a sequence that you define.

You can assign variables or signals to the wiring test. The following options are available:

  • Enter the signal in the dialog

  • Select a variable from the list of internal variables (e.g. system variables)

  • Read in PLC signals from the TIA Portal and select signals

Setting areas of the individual safety functions

An acceptance test can be carried out for every safety function used on the machine. Although the list in the window shows all of the safety functions, it is possible that not all safety functions are used on a particular machine. In this case, the affected part of the acceptance test (test type) for the unused safety function can be deactivated (see "Configuring acceptance tests").

When testing a safety function (e.g. SOS), the response of the safe monitoring function is checked, and it is recorded if individual function tolerances are exceeded.

Example: An axis is traversed. The test then checks how quickly the safety function responded and to what extent the axis still moved outside the tolerance window.

You can make the following settings here:

  • Define a test name

  • Conditions for triggering the test

  • Acquired alarms that should appear in the report

  • Confirm the test result

The following safety functions can be used as test types:

SI function

Explanation

STO

The test verifies the following points for an axis/drive:

  • Principle of operation of the hardware

  • Wiring of the switch-off signal paths (only for Basic)

  • Assignment of the terminals for STO on the Control Unit (only for Basic)

  • STO grouping (if available)

  • Parameterizing the STO function

In addition, the following points are verified if Safe Brake Control (SBC) is configured for the axis/drive:

  • Connecting the brake

  • Principle of operation of the hardware

  • Parameterizing the SBC function

  • Forced checking procedure for the brake control

SS1

The test verifies the following points for an axis/drive:

  • Principle of operation of the hardware

  • Wiring of the switch-off signal paths (only for Basic)

  • Assignment of the terminals for SS1 on the Control Unit (only for Basic)

  • SS1 grouping (if available)

  • Parameterizing the SS1 function

In addition, the following points are verified if Safe Brake Control (SBC) is configured for the axis/drive:

  • Connecting the brake

  • Principle of operation of the hardware

  • Parameterizing the SBC function

  • Forced checking procedure for the brake control

SS2

This test verifies the following points:

  • Initiating Safe Stop 2 (SS2) is detected

  • The associated axis/drive brakes

  • Monitoring for Safe Operating Stop (SOS) for this axis/drive is activated when the parameterized transition time elapses

SS2E

This test verifies the following points for an axis/drive:

  • Triggering of Safe Stop 2 External (SS2E) is detected

  • The associated axis/drive brakes

  • Monitoring for Safe Operating Stop (SOS) for this axis/drive is activated, or a safe stop (STO) is initiated by the application once the parameterized transition time has elapsed

SS2ESR

This test verifies the following points for an axis/drive:

  • Triggering of Safe Stop 2 Extended Stop and Retract (SS2ESR) is detected

  • The associated axis/drive brakes

  • Monitoring for Safe Operating Stop (SOS) for this axis/drive is activated, or a safe stop (STO) is initiated by the application once the parameterized transition time has elapsed

SOS

When the standstill tolerance for Safe Operating Stop (SOS) is violated, the test verifies whether the following points occur:

  • The drive is stopped

  • Stop responses and alarms of both monitoring channels are initiated

  • Response times and over-travel comply with the requirements

SLS

This test checks whether an overshoot of the Safely Limited Speed (SLS) is detected.

If a configured SLS limit value is exceeded, the configured stop responses and alarms should be triggered for both monitoring channels. The response times and over-travel should comply with the requirements.

For SLS limit value 1, an override can additionally be specified via the fail-safe PLC.

SSM

This test verifies the following points for an axis/drive:

  • It is detected if the limit value of the Safe Speed Monitor is exceeded

  • A message is output

SLA

This test verifies the following points:

  • Unwanted increase in speed is detected

  • The associated axis/drive brakes

  • Stop response (configured in p9579) is activated for the axis/drive

SLP

This test verifies the following points:

  • It is detected if the upper or lower limit value of the Safely-Limited Position (SLP) is violated

  • The monitoring limit values for SLP correspond with the required machine positions

  • The configured stop responses and alarms of both monitoring channels are initiated when an SLP limit value is violated (exceeded or fallen below)

SBT

Verifies whether the brake is functioning correctly and that the required holding torque is observed.

SCA

This test must be performed separately for each axis/drive. Plus and minus cams must be checked independently of each other.

SDI

Testing the Safe Direction (SDI) verifies for an axis/drive whether the following points occur when the configured SDI tolerance is exceeded in the blocked direction:

  • The drive is stopped

  • Configured stop responses and alarms of both monitoring channels are initiated

  • Response times and over-travel comply with the requirements

After the acceptance test, the alarms that occurred are listed and a trace generated with the relevant signals. You must then subsequently check the list as to whether all of the expected alarms have occurred, and whether the results in the trace recording are correct.
